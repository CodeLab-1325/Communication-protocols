# 2단계: HTTP/1.1 (웹 통신의 표준)

이 폴더에서는 FastAPI나 Django, Flask 같은 **어떠한 웹 프레임워크도 사용하지 않고**, 파이썬 내장 `socket` 모듈(1단계 TCP 통신)만을 이용하여 **날것의 HTTP 서버(Raw HTTP Server)**를 밑바닥부터 구현해 봅니다.

## 🎯 학습 목표
우리가 매일 쓰는 웹 브라우저가 내부적으로는 결국 "어떤 문자열(텍스트)을 TCP 소켓을 통해 주고받고 있는지" 그 실체를 눈으로 확인합니다.
또한, 프레임워크가 대신 해주던 **파싱(Parsing) - 라우팅(Routing) - 직렬화(Serialization)** 과정을 직접 코딩하며 '레이어드 아키텍처'의 필요성을 체감합니다.

---

## 💻 실행 방법

### 웹 서버 실행 및 접속 테스트
**터미널 (서버 실행):**
```bash
python stage2-http1.1/server.py
```

서버가 실행되면, 웹 브라우저(크롬, 사파리 등)를 열고 주소창에 아래 URL을 입력하세요.
- 👉 **http://localhost:8080/**

또한 API 응답 테스트를 위해 아래 주소도 들어가 보세요.
- 👉 **http://localhost:8080/api/data**

접속 후 **터미널 로그**를 확인해 보세요. 브라우저가 화면을 그리기 위해 뒤에서 몰래 서버로 보낸 HTTP 헤더 정보들이 파싱되어 찍혀 있을 것입니다!

## 🔍 핵심 관전 포인트 (아키텍처 분리)

1단계에서는 모든 로직이 `server.py` 하나에 뭉쳐 있었지만, 이번에는 역할별로 파일을 나누었습니다. 웹 프레임워크들이 내부적으로 가지고 있는 핵심 모듈들입니다.

1. **`http_parser.py` (파싱 계층)**
   - 브라우저가 보낸 `GET / HTTP/1.1\r\nHost: ...` 와 같은 거친 텍스트 덩어리를 예쁜 파이썬 객체(`HttpRequest`)로 분해해 줍니다.
2. **`router.py` (라우터)**
   - 들어온 요청의 주소(URL Path)를 보고, 어떤 함수를 실행할지 길을 안내합니다. FastAPI의 `@app.get("/")` 데코레이터가 내부적으로 하는 역할입니다.
3. **`handlers.py` (컨트롤러/비즈니스 계층)**
   - 실제 HTML 텍스트를 만들거나 JSON 데이터를 만들어내는 실제 '일'을 하는 곳입니다.
4. **`http_response.py` (직렬화 계층)**
   - 비즈니스 계층에서 만든 파이썬 객체를, 다시 브라우저가 알아들을 수 있는 문자열 형태로 조립해서 반환해 줍니다.
